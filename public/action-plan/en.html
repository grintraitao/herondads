Takeaway. Acquisition is both shrinking and quality-skewed. The lever is not more installs but concentrating UA (user acquisition) on core Indonesia, testing scale in the high-quality small markets (Philippines, Egypt, Vietnam), and fixing localization for low-retention languages.

06 · Problems

Five concrete problems

1. No active acquisition engine

88 percent of users come from Google Play organic, almost no paid or referral. When organic stalls, there is no lever to pull up.

2. Wrong users in the wallet area

Reward drops pulled in reward hunters and bots, used once then gone, never real scale.

3. The activation wall

About 74 percent of new users leave after one day, over half uninstall the same day. The common thread is no feature activation. There is a proven lever: day-0 activation lifts return from 28 to 45 percent.

4. The core is not in Web3

Those who stay do so for the browsing utility, not the wallet. The wallet is barely touched, W4 only 6 to 7 percent.

5. No revenue and weak measurement

No revenue model yet. Missing user_id and session_id on many events makes funnel and retention only a floor. Core value such as ad blocking and speed is barely tracked. The push channel for re-engagement sits unused.

Root cause. The core value is not enough to both pull users in and keep them. Resources go to the wallet and Web3, which the core group barely uses; while the browsing utility, the real value, has not been deepened.

08 · Optimizations to debate

Four changes aimed at acquisition, retention and Herond Point

These are hypotheses to test, not conclusions. Each proposal ties to a problem in the data, with trade-offs and how to measure for A/B.

1. Streamlined onboarding: from three screens to one, with and without sign-in variantsHypothesis to test
Why (from data)
The funnel drops 51 percent at Welcome to Login and 53 percent at OTP verification; first-run desktop only 22 percent set default. The more onboarding steps, the more users fall off.
Trade-offs and risks
Cutting steps may lower the sign-in or default-set rate, so measure the offset via activation and retention. The concrete screens and flow will be shared for debate.
How to measure
A/B the one-screen version vs current; measure onboarding completion, day-0 activation, and D1, D7 retention.
2. Remove the wallet button from the nav bar, hide it in the menuHypothesis to test
Why (from data)
The wallet is nearly inactive (about 20 wallets sign a transaction per month, only 0.8 percent of users touch it) but occupies prime nav space. The retention core is the browsing utility, not Web3.
Trade-offs and risks
Real Web3 users need one more step to open the wallet, but this group is very small; the wallet should still surface contextually when entering a dApp. Prime space should go to a value action like check-in.
How to measure
Compare wallet opens and retention before and after; measure usage of whatever feature takes the replaced slot.
3. Herond Point without required sign-in: check-in and points right on the New TabHypothesis to test
Why (from data)
0 percent of mobile users sign in, so all earn value never reaches the largest group. Day-0 activation lifts return from 28 to 45 percent but only 19 percent do it. Login-free earn turns check-in into an easy day-one activation.
Mechanism
Per the server-authoritative design: the client sends events not points; the server issues the anon-id, looks up the rule and rate-limits; on sign-in it merges anon points into the real account; redeem only after sign-in. So editing local files cannot add points.
Trade-offs and risks
Opening anonymous earn makes multi-account farming cheap; mitigate with a pending-points cap, crediting only after sign-in (pending then confirm), and device fingerprint. Needs added backend: anon account, points ledger, earn rule.
How to measure
Day-0 check-in rate, day-0 activation, D1 and D7 retention, and the sign-in rate after earning (merge).
4. Optimize new-user acquisition by quality, not spray-and-pray installsHypothesis to test
Why (from data)
New-user quality is very skewed by segment. Indonesia is 36 percent of users retained to D7 (D7 8.4); India has large install but D7 only 5.9 so it burns installs with few lasting; Philippines, Egypt, Vietnam have high D7 (7.6 to 10.9) but are small. 98 percent of the base is budget Android.
How
  • Concentrate UA and ASO (app store optimization) on Indonesia, invest in Indonesian localization (highest D7 9.1).
  • Test scaling Philippines, Egypt, Vietnam: raise volume and watch whether D7 holds.
  • Re-localize for low-retention languages (Spanish, Portuguese, Arabic) or de-prioritize.
  • Measure cold start and crash on the largest budget-device group (Redmi, Galaxy A) to learn whether performance is a reason for leaving, then decide on investment.
Trade-offs and risks
Concentrating on Indonesia raises single-market dependence. Scaling small markets may pull quality down as volume rises, so test step by step.
How to measure
Cost per user retained to D7 (not per install), and D7 by country and language before and after.

11 · Decision gates, risks, validation

When to continue, when to change course

Mid-term gate, month 3. If D1 and D7 retention of new cohorts clearly improve over baseline, and the day-0 check-in rate rises, continue.
Final gate, month 6. If real-user retention rises sustainably, double down. If not, revisit product positioning rather than fighting Chrome as a browser.

What must be true

  • Instrumentation is fixed, reliable enough to read the numbers.
  • Streamlined onboarding genuinely lifts day-0 activation.
  • Login-free earn is embraced by users without breaking from farming.
  • The core browsing value gets deepened.
Main risks. If instrumentation is not fixed, every conclusion is wrong. Opening anonymous earn can be farmed without caps and controls. Onboarding changes may lower the sign-in rate, so the offset must be measured via activation.

90-day validation

Run an A/B of streamlined onboarding and login-free earn on part of the user base. Measure day-0 activation, D1 and D7 retention, check-in rate, and merge rate after sign-in. Pass the gate when activation and retention clearly improve over the control group.

Glossary

Specialized terms used in this document

RetentionShare of users returning after a number of days.
ChurnUsers who leave and are no longer active in the period.
ActivationA new user completing a first value action (bookmark, download content, sign in).
OnboardingThe intro and setup screens on first app open.
CohortA group of users sharing a milestone, e.g. same first-open month, tracked over time.
MAU and DAUUsers active in 30 days (MAU) and in one day (DAU).
D1, D7, D30Return rate after 1, 7, 30 days from first open.
ResurrectedLong-dormant users coming back.
Right-censoredA cohort too young to be fully observed, so the number is understated.
FunnelA step-by-step conversion path, e.g. the wallet-creation flow.
UA (user acquisition)The work and budget to bring in new users.
Opt-inThe user actively turns on or agrees to participate.
